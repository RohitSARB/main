## 2️⃣ What is `@app.get("/")` ?

This line is **the most important concept here**.

```python
@app.get("/")
```

### 🔹 What `@` means (in Python)

`@` is called a **decorator**.

A decorator:

* is a function that **modifies another function**
* runs **at definition time**, not when the function is called

So this:

```python
@app.get("/")
def root():
    return {"message": "FastAPI is running"}
```

Is roughly equivalent to:

```python
def root():
    return {"message": "FastAPI is running"}

app.get("/")(root)
```

👉 `app.get("/")` **registers** `root()` as a handler for:

* HTTP **GET**
* path **"/"**

---

## 3️⃣ What happens when a request comes in?

When someone opens:

```
http://127.0.0.1:8000/
```

FastAPI:

1. Sees a **GET** request to `/`
2. Finds the function registered for that route
3. Calls `root()`
4. Converts the returned dictionary into **JSON**
5. Sends it back as the HTTP response

So this:

```python
return {"message": "FastAPI is running"}
```

Becomes:

```json
{
  "message": "FastAPI is running"
}
```

## 4️⃣ What happens if you write multiple functions?

Totally normal and expected 👍

Example:

```python
@app.get("/")
def root():
    return {"message": "Home"}

@app.get("/users")
def get_users():
    return ["Alice", "Bob"]

@app.get("/health")
def health():
    return {"status": "ok"}
```

Now you have:

* `/` → `root()`
* `/users` → `get_users()`
* `/health` → `health()`

Each function handles **one route**.

This means:

* GET `/hello` → `hello()`
* POST `/hello` → `hello()`

Same function, multiple HTTP methods.

### 🔹 Case 2: Multiple FastAPI decorators for the same path

```python

@app.post("/items")

    return {"message": "Item created"}
```

Same path, **different methods**, **different functions**.

This is very common in APIs.

Let’s strip the magic away and look at what `Query` *actually is* and *how FastAPI uses it*.

---

## What `Query` actually is

```python
from fastapi import Query
```

👉 **`Query` is a function that returns a special metadata object**
That object tells FastAPI:

> “This parameter comes from the **query string**, and here are the rules for it.”

You are **not calling `Query` at runtime**.
FastAPI reads it **when the app starts**, inspects your function signature, and builds:

* Validation rules
* OpenAPI schema
* Swagger UI inputs
* Error handling

---

## Without `Query` (pure Python)

```python
def search_items(q: str, limit: int = 10):
    ...
```

Python sees:

* `q` → required argument
* `limit` → optional argument with default `10`

FastAPI *infers*:

* These must be **query parameters**
* Validation = only type checking

---

## With `Query` (FastAPI metadata)

```python
def search_items(
    q: str = Query(..., min_length=2),
    limit: int = Query(10, ge=1, le=100)
):
    ...
```

Python sees:

* Default values are **not ints or strings**
* They’re **special objects**

FastAPI sees:

* “Ah, this param uses `Query`”
* “Apply these validation rules”
* “Expose them in OpenAPI”

---

## Concrete mental model (this is the key 🧠)

Think of `Query` as a **label + rulebook** attached to a parameter.

```text
q: str
│
├── comes from query string
├── required
├── min length = 2
├── show in Swagger
└── auto-validate
```

---

## End-to-end example (what actually happens)

### Code

```python
from fastapi import FastAPI, Query

app = FastAPI()

@app.get("/search")
def search_items(
    q: str = Query(..., min_length=3),
    limit: int = Query(10, ge=1, le=5)
):
    return {"q": q, "limit": limit}
```

---

### Request 1 (valid)

```
GET /search?q=fastapi&limit=3
```

FastAPI does:

1. Extracts `q` and `limit` from URL
2. Validates:

   * `len("fastapi") >= 3` ✅
   * `1 ≤ 3 ≤ 5` ✅
3. Calls your function:

   ```python
   search_items(q="fastapi", limit=3)
   ```
4. Returns response

✅ **Your function only runs if validation passes**

---

### Request 2 (invalid)

```
GET /search?q=hi&limit=10
```

Validation fails:

* `"hi"` too short
* `10 > 5`

FastAPI:

* ❌ Does NOT call your function
* ✅ Returns automatic error response:

```json
{
  "detail": [
    {
      "loc": ["query", "q"],
      "msg": "ensure this value has at least 3 characters",
      "type": "value_error.any_str.min_length"
    },
    {
      "loc": ["query", "limit"],
      "msg": "ensure this value is less than or equal to 5",
      "type": "value_error.number.not_le"
    }
  ]
}
```

---

## Why this is powerful (real-world use)

### Pagination

```python
@app.get("/items")
def list_items(
    page: int = Query(1, ge=1),
    size: int = Query(10, le=100)
):
    offset = (page - 1) * size
    return {"page": page, "size": size, "offset": offset}
```

### Search filters

```python
@app.get("/products")
def products(
    q: str | None = Query(None, min_length=2),
    min_price: float = Query(0, ge=0),
    max_price: float = Query(10000)
):
    ...
```

---

## One-liner takeaway

> `Query` is **declarative validation + documentation metadata** that FastAPI reads *before* your function ever runs.

No magic.
No runtime cost.
Huge clarity.

## What a path parameter actually is

```python
@app.get("/users/{user_id}")
def get_user(user_id: int):
    return {"user_id": user_id}
```

The part inside `{}` is a **path parameter**.

It means:

> “This value is part of the URL path itself, not after `?`.”

---

## What happens with this request

### Request

```
/users/10
```

FastAPI does this internally:

```python
user_id = 10
```

Then it calls your function:

```python
get_user(user_id=10)
```

### Response

```json
{
  "user_id": 10
}
```

---

## How FastAPI knows what `10` is

Because of this line:

```python
def get_user(user_id: int):
```

FastAPI:

* Extracts `"10"` from the URL
* Converts it to `int`
* Validates it

If you try:

```
/users/abc
```

You’ll get:

```json
{
  "detail": [
    {
      "loc": ["path", "user_id"],
      "msg": "value is not a valid integer",
      "type": "type_error.integer"
    }
  ]
}
```

Your function is **never called**.

---

## Where path params are used (real meaning)

Path parameters identify **a specific resource**.

Think:

| URL                | Meaning           |
| ------------------ | ----------------- |
| `/users`           | All users         |
| `/users/10`        | User with ID = 10 |
| `/orders/55`       | Order #55         |
| `/products/abc123` | Product abc123    |

Path = **identity**
Query = **options / filters**

## Make it real using YOUR users list

Right now your users look like:

```python
users = [
    {"name": "rohit", "age": 1},
    {"name": "rohit", "age": 2},
    {"name": "sharma", "age": 2}
]
```

Let’s pretend index = ID.

```python
@app.get("/users/{user_id}")
def get_user(user_id: int):
    if user_id < 0 or user_id >= len(users):
        return {"error": "User not found"}

    return users[user_id]
```

### Request

```
/users/1
```

### Response

```json
{
  "name": "rohit",
  "age": 2
}
```

## Path vs Query (this matters)

### ❌ Bad API design

```
/users?id=10
```

### ✅ Good REST design

```
/users/10
```

Because:

* Path params = **which thing**
* Query params = **how / options**

## 🧠 The Real Rule (slightly refined)

> **Use `async def` by default.**
> Use `def` **only** when the code inside is *purely blocking* and you **cannot** make it async.

FastAPI is **async-first**, but it’s also **sync-compatible** on purpose.

---

## ⚙️ What actually happens under the hood

### `def` (sync)

```python
@app.get("/sync")
def sync_endpoint():
    return {"type": "sync"}
```

* Runs in a **threadpool**
* One request = one worker thread
* Blocks that thread until finished
* Scales poorly for I/O-heavy work

---

### `async def` (async)

```python
@app.get("/async")
async def async_endpoint():
    return {"type": "async"}
```

* Runs on the **event loop**
* Can pause on `await`
* Frees the worker to handle other requests
* Scales to **thousands of concurrent connections**

---

## 🌍 Why async matters (real world)

Async shines when you **wait** on something:

✅ Database queries
✅ HTTP calls to other services
✅ Redis / Kafka / S3
✅ Web scraping
✅ File I/O

```python
@app.get("/users")
async def get_users():
    users = await db.fetch_all("SELECT * FROM users")
    return users
```

While the DB is working, FastAPI handles *other requests*.
That’s the superpower 💥

---

## 🚨 Common Trap (very important)

❌ **Blocking code inside `async def`**

```python
import time

@app.get("/bad")
async def bad():
    time.sleep(5)  # ❌ blocks the event loop
    return {"msg": "oops"}
```

This **kills performance**.

✅ Correct async version:

```python
import asyncio

@app.get("/good")
async def good():
    await asyncio.sleep(5)
    return {"msg": "ok"}
```

---

## 🧩 When `def` is actually OK

Use **sync `def`** if:

* Library has **no async support**
* CPU-bound work (image processing, ML inference)
* Legacy code

FastAPI will safely offload it:

```python
@app.get("/hash")
def hash_password():
    return bcrypt.hashpw(b"secret", bcrypt.gensalt())
```

---

## 🏆 Golden Rules (pin this)

* **Default** → `async def`
* **I/O work** → async + `await`
* **CPU-heavy / legacy** → `def`
* ❌ Never block inside `async def`
* FastAPI ≠ magically async — *you* must use async libraries

## 1️⃣ What `APIRouter` does (in plain English)

`APIRouter` is a **mini FastAPI app**.

It lets you:

* Group related endpoints (here: *tasks*)
* Keep code modular & readable
* Plug routes into the main app later

Think of it like this:

> **FastAPI app = root**
> **APIRouter = feature module**

---

## 2️⃣ This line is the key

```python
router = APIRouter(prefix="/tasks", tags=["Tasks"])
```

### Meaning:

* `prefix="/tasks"`
  👉 Every endpoint in this router **starts with `/tasks`**

* `tags=["Tasks"]`
  👉 In Swagger UI, these endpoints appear under **Tasks** section

---

## 3️⃣ How router connects to the main app

Somewhere in `main.py` you’ll have:

```python
from fastapi import FastAPI
from app.routes.task import router as task_router

app = FastAPI()
app.include_router(task_router)
```

Once included:

* Your routes become **live**
* FastAPI knows about them

---

## 4️⃣ Final URLs that exist

Given your router:

```python
@router.post("/")
@router.get("/")
```

Because of the prefix `/tasks`, the actual URLs are:

| Method | URL      |
| ------ | -------- |
| POST   | `/tasks` |
| GET    | `/tasks` |

---

## 5️⃣ POST `/tasks` — What happens & output

### Request body (example)

```json
{
  "title": "Learn FastAPI",
  "description": "Router concept",
  "priority": 1
}
```

### Flow

1. Request hits `/tasks`
2. FastAPI validates input using `TaskCreate`
3. Data is inserted into MongoDB
4. Mongo generates `_id`
5. You convert `_id` → `id` (string)
6. Response is validated by `TaskResponse`

### Response (output)

```json
{
  "id": "65f1a2c9e8b3a91234abcd90",
  "title": "Learn FastAPI",
  "description": "Router concept",
  "priority": 1
}
```

💡 `_id` never leaks — clean API 👌

---

## 6️⃣ GET `/tasks` — What happens & output

### Flow

1. Mongo returns a cursor
2. `async for` loops over documents
3. Each document is transformed
4. `_id` → `id`
5. Response validated against `list[TaskResponse]`

### Response (output)

```json
[
  {
    "id": "65f1a2c9e8b3a91234abcd90",
    "title": "Learn FastAPI",
    "description": "Router concept",
    "priority": 1

    "id": "65f1a2c9e8b3a91234abcd91",
```
